SHL Conversational Assessment Recommender

Approach Document  ·  AI Intern Take-Home  ·  2 July 2026

Problem Framing

Hiring managers arrive with vague intent ("I need something for a Java developer") and no knowledge of SHL vocabulary. The goal is a stateless conversational agent that navigates from that intent to a grounded shortlist through multi-turn dialogue — clarifying when needed, refining on new constraints, comparing on demand, and refusing everything outside the SHL Individual Test Solutions catalog.

Architecture

Guard→ Planner→ Clarifier→ Decision→ Hybrid Retrieval→ Context Builder→ LLM (Groq)→ Verifier→ ChatResponse

The API is fully stateless: every POST /chat receives the complete conversation history and rebuilds all state from scratch. No session store is needed, which simplifies deployment and makes the system trivially horizontally scalable.

  • Guard Engine — runs first; blocks prompt injection (15 regex patterns), retrieval manipulation, off-topic queries (40+ patterns), legal questions, and oversized inputs (>5000 chars).
  • Planner — extracts structured slots (role, seniority, language, duration, skills, categories, comparison items) from every user turn across the full history.
  • Clarifier — enforces that role is present before retrieval. All other slots are optional.
  • Decision Engine — routes to one of: ASK_CLARIFICATION, RETRIEVE, COMPARE, GREETING, SHOW_HELP, REJECT, UNKNOWN.
  • Turn Cap — at message 7 (response = turn 8, the evaluator limit) force_recommendation=True bypasses clarification and non-retrieval branches, guaranteeing a shortlist on the last allowed turn.
  • Verifier — post-generation: flags hallucinated names, invalid URLs, and duplicate items.

Retrieval Setup

The SHL Individual Test Solutions catalog was scraped and normalized into data/processed/knowledge_base.json. Each document stores name, description, URL, duration, languages, job levels, categories, remote flag, and a search_text field (concatenation of all fields) used for indexing.

Prompt Design

Three prompt templates are used: RECOMMEND, COMPARE, and REFINE. All share a system prompt with ten strict grounding rules ("never invent assessment names / URLs / durations") and a 120-word reply cap to stay within the 30-second per-call timeout. The LLM (Groq llama-3.3-70b-versatile, temperature 0.2) is only asked to explain the retrieved assessments, never to select them — selection is done entirely by the retrieval + filter + ranker stack. The COMPARE template instructs the model to produce a grounded diff from context only and returns an empty recommendations array (informational turn, not a committed shortlist).

Agent Design Decisions

  • When to clarify: only when role is absent. Asking for every optional slot (seniority, language, duration) before retrieving proved too slow and annoyed users who had already given a job description.
  • When to recommend: once role is known, retrieve immediately and let the user refine rather than front-loading questions.
  • When to compare: whenever the message contains "compare", "difference", "vs", "versus", or "better than" AND at least two catalog items can be resolved from the message.
  • When to end: end_of_conversation: true is set only when a shortlist already existed on the prior turn and the current turn added no new slots (confirmation pattern) — or when the turn cap forces a final answer.
  • Refinement: free text from all user turns is accumulated for BM25/FAISS, while structured slots accumulate additively (skills/categories are union-merged, other scalars are overwritten by newer values).

Evaluation Approach

Evaluation ran at three levels:

LevelMethodSignal
Unit 40+ pytest modules covering every component (guards, planner, retrieval, generation) Regression safety net; fails fast on schema violations
Trace replay Manual replay of all 10 public conversation traces; checked schema, recommendations, and end-of-conversation flags at every turn Approximates Recall@10 and behavior-probe pass-rate
Adversarial Direct injection of prompt-injection strings, off-topic queries, and edge inputs against the running service Guard coverage and refusal consistency

Improvement was measured by tracking which trace turns changed from "wrong behavior" to "correct behavior" after each fix, using the DEBUG_NOTES as a structured regression log. Recall@10 was estimated by counting catalog-correct items in the top-10 shortlist across the 10 traces before and after retrieval changes.

What Didn't Work and How It Was Fixed

Four critical bugs were found through trace replay and fixed iteratively — none were caught by unit tests alone, which underscores the value of end-to-end trace evaluation.

Multi-turn context loss Role extracted on turn 1 was gone by turn 3; agent re-asked the same question Preserve full transcript roles in _messages_from_chat_request; accumulate all user turns for free_text

Stack & Tooling

Runtime: FastAPI + Uvicorn  ·  LLM: Groq llama-3.3-70b-versatile via OpenAI-compatible SDK (free tier, ~1 s latency)  ·  Embeddings: sentence-transformers/all-MiniLM-L6-v2  ·  Vector store: FAISS CPU  ·  Keyword index: BM25 (rank-bm25)  ·  Deploy: Docker + Render (free tier)  ·  AI tools: GitHub Copilot used for boilerplate; all architecture, prompt, and agent design decisions were made and can be defended independently.
